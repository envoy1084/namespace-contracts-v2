## Verification Workflow

For each issue:

1. Write a temporary proof test that demonstrates the issue against the current code.
2. Run the proof test and record whether the issue is real.
3. If real, implement the smallest production fix that preserves the architecture.
4. Keep or convert the proof into a regression test.
5. Run focused tests, then broader verification.
6. Commit each meaningful change using a conventional commit message.

## Detailed Findings

### C-01: Malformed UTF-8 can pass emoji-only rules

Status: Fixed.

`LabelClassRule._nextCodepoint` decodes multibyte codepoints by masking each continuation byte with `0x3F`, but it does not verify that continuation bytes are actually in the UTF-8 continuation range `0x80..0xBF`. A malformed label can therefore be interpreted as a valid emoji codepoint by the rule engine even though it is not valid UTF-8.

Expected validation:

- Proof: `test_evaluateMint_TEMP_malformedUtf8BypassesEmojiRequirement` passed before the fix, showing malformed bytes `0xF0 0x5F 0x58 0x40` were priced as an emoji label.
- Regression: `test_evaluateMint_rejectsMalformedUtf8ThatWouldDecodeAsEmoji` now expects `InvalidUtf8Label`.
- Fix: `_nextCodepoint` now validates continuation bytes, rejects overlong encodings, rejects surrogate ranges, and rejects values above `U+10FFFF`.

### C-02: Renewal can use caller-selected activation for labels not governed by that activation

Status: Fixed.

`NamespaceController.renew` loads state from the activation registry and only rejects `AVAILABLE`. This allows two related paths:

- `RESERVED` labels can be renewed because ENSv2 `PermissionedRegistry.renew` accepts registered or reserved labels.
- If multiple activations share a registry, a caller can use a cheaper or less restrictive activation to renew a label originally minted under a stricter activation.

Expected validation:

- Proof: `test_renew_TEMP_extendsReservedLabel` passed before the fix, showing a `RESERVED` label expiry could be extended through the controller.
- Proof: `test_renew_TEMP_usesCheaperActivationForLabelMintedElsewhere` passed before the fix, showing a free activation could renew a paid-activation label without additional payment.
- Regression: `test_renew_revertsForReservedLabel` now expects `LabelNotRenewable` for `RESERVED` labels.
- Regression: `test_renew_revertsWhenActivationDidNotMintLabel` now expects `LabelActivationMismatch`.
- Fix: mints now record `labelActivations[registry][labelHash] = activationId`; renewals require `REGISTERED` status and a matching activation binding.

### C-03: Revoking registry admin does not stop active mint/renew

Status: Fixed.

Activation lifecycle functions re-check that `activation.owner` still has root `ROLE_REGISTRAR_ADMIN`. Public `mint` and `renew` only check `activation.active`. If the owner loses registry admin rights while the controller still has registry mint/renew roles, the activation can remain live and unusable by the new admin except through global controller role revocation.

Expected validation:

- Proof: `test_mint_TEMP_succeedsWhenActivationOwnerLostRegistryAdmin` and `test_renew_TEMP_succeedsWhenActivationOwnerLostRegistryAdmin` passed before the fix.
- Regression: `test_mint_revertsWhenActivationOwnerLostRegistryAdmin` and `test_renew_revertsWhenActivationOwnerLostRegistryAdmin` now expect `UnauthorizedActivationOwner`.
- Fix: `mint` and `renew` now call `_checkRegistryAdminAuthority(activation.owner, activation.registry)` before evaluating runtime data or touching registry state.

### C-04: Flat pricing allows near-permanent names for one payment

Status: Fixed.

`duration` is public caller input. `FixedPriceRule` and `USDOracleRule` can return a flat amount independent of duration, and the controller does not enforce a maximum duration. A buyer can choose a very large duration and receive a near-permanent expiry for the same configured price.

Expected validation:

- Proof: `test_mint_TEMP_flatPriceAllowsNearPermanentRegistration` passed before the fix, minting to `type(uint64).max` expiry for the normal fixed price.
- Regression: `test_mint_revertsWhenDurationExceedsActivationMax` now expects `DurationOutOfBounds`.
- Fix: activation config now includes `minDuration` and required `maxDuration`; activation creation validates the bounds, and `mint`/`renew` enforce them before pricing or registry calls.

### C-05: Revoked activation owner can still pause or unpause

Status: Fixed.

`PauseRule.setPaused` checks only `msg.sender == activation.owner`. It does not verify that the owner still has registry admin authority. This can allow a revoked owner to pause or unpause traffic after the registry owner intended to remove their authority.

Expected validation:

- Proof: `test_setPaused_TEMP_allowsActivationOwnerAfterRegistryAdminRevoked` passed before the fix.
- Regression: `test_setPaused_revertsWhenActivationOwnerLostRegistryAdmin` now expects `UnauthorizedActivationOwner`.
- Fix: `PauseRule.setPaused` now requires the activation owner to still hold root `ROLE_REGISTRAR_ADMIN` on the activation registry.

### C-06: Fee-on-transfer ERC20s underpay while mint/renew succeeds

Status: Fixed.

`ERC20PaymentModule` and `ERC20SplitPaymentModule` trust nominal `safeTransferFrom` success. Fee-on-transfer tokens can make the recipient receive less than `price.amount`, while the mint or renew operation still succeeds.

Expected validation:

- Proof: `test_collectMint_TEMP_succeedsWhenFeeOnTransferTokenUnderpaysRecipient` and `test_collectMint_TEMP_succeedsWhenFeeOnTransferTokenUnderpaysRecipients` passed before the fix.
- Regression: both tests now expect `PaymentAmountMismatch`.
- Fix: both payment modules now compare recipient balance deltas against the expected transfer amount and reject fee-on-transfer or otherwise non-exact ERC20 behavior.

### C-07: ERC20 balance gates are flash-loanable

Status: Fixed.

`TokenBalanceRule._evaluate` checks a live ERC20 balance. If the configured token is borrowable or otherwise temporarily transferable, a buyer can satisfy eligibility or receive discounts without durable ownership.

Expected validation:

- Proof: `test_tokenBalance_TEMP_temporaryBalancePassesEligibility` passed before the fix.
- Regression: `test_tokenBalance_revertsWhenHoldTimeNotMet` now expects `TokenBalanceHoldTimeNotMet`.
- Fix: `TokenBalanceRule` now requires a non-zero `minHoldTime` when `minBalance` is set. Users call `recordBalance`, then must still hold the balance after the hold period before mint/renew evaluation passes.

### C-08: Registrar-admin-only owners can monetize renewals

Status: Fixed.

Activation creation checks only `ROLE_REGISTRAR_ADMIN`, while the controller itself needs `ROLE_REGISTRAR | ROLE_RENEW` on the registry. An account with registrar-admin authority but not renew-admin authority may configure renewal pricing and revenue collection through controller-held `ROLE_RENEW`.

Expected validation:

- Proof: `test_activate_TEMP_succeedsWhenOwnerLacksRenewAdmin` passed before the fix.
- Regression: `test_activate_revertsWhenCallerLacksRenewAdmin` now expects `UnauthorizedActivationOwner`.
- Fix: the shared registry-admin authority check now requires both `ROLE_REGISTRAR_ADMIN` and `ROLE_RENEW_ADMIN`; `PauseRule` uses the same authority set for pause control.

## Detailed Leads

### L-01: Unbound resolver node writes can target unrelated resolver nodes

Status: Fixed.

The hooks compute resolver nodes from activation-provided `parentNode` and label hash. If an activation can set an unrelated `parentNode` and the hook has broad resolver write authority, minting in one registry could write records for another namespace's resolver node.

- Proof: `test_activate_TEMP_allowsMismatchedParentNode` passed before the fix, showing an activation could store `victim.eth` as `parentNode` while using Alice's registry.
- Regression: `test_activate_revertsWhenParentNodeDoesNotMatchRegistry` now expects `RegistryParentNodeMismatch`; `test_activate_revertsWhenRegistryChainDoesNotReachConfiguredRoot` rejects a fake self-consistent registry tree.
- Fix: activation now verifies the configured `parentNode` against the registry's canonical ENS v2 parent chain, anchored to the controller's configured root registry, via `getParent()` and `getSubregistry(label)` before storing the activation.

### L-02: Module approval revocation does not quarantine existing activations

Status: Fixed.

Module approval is checked when storing activation module lists. Existing activations continue executing stored modules, and `updateModuleConfig` does not re-check approval. This may be intentional compatibility, but it is not an emergency-disable mechanism.

- Proof: `test_updateModuleConfig_TEMP_allowsRevokedRuleModule` and `test_mint_TEMP_executesRevokedRuleModule` passed before the fix.
- Regression: `test_updateModuleConfig_revertsWhenRuleModuleRevoked`, `test_mint_revertsWhenRuleModuleRevoked`, `test_mint_revertsWhenPaymentModuleRevoked`, and `test_mint_revertsWhenPostHookModuleRevoked` now expect `UnapprovedModule`.
- Fix: runtime rule evaluation, payment collection, post-hook execution, and module config updates now re-check module approval when `moduleApprovalRequired` is enabled.

### L-03: Overlapping reservation claims let callers choose the least restrictive proof

Status: Confirmed design limitation.

`ReservationRule` verifies one caller-supplied claim. If the same root contains multiple valid claims for one label, callers can choose the claim that is most favorable. The root generation process must enforce uniqueness or the contract must encode precedence.

- Proof: `test_reservation_TEMP_overlappingClaimsCanChooseExpiredPermissiveProof` passed, showing a root with both a blocking claim and an expired permissive claim lets the caller submit the expired proof and avoid the active block.
- Fixability: not safely fixable inside the current unordered Merkle-inclusion design. The contract receives only one claim and one proof, so it cannot prove that no stricter sibling claim for the same label exists elsewhere in the root.
- Required design change: replace the reservation root with a keyed uniqueness proof, such as a sparse Merkle map keyed by `labelHash`, or another commitment where each label has exactly one authoritative reservation record. Until then, activation operators must treat "one claim per label per root" as a hard off-chain tree-generation invariant.

### L-04: Native pricing is documented but bundled payment modules reject native ETH

Status: Fixed.

Several rule docs described `token = address(0)` as native ETH pricing, but the bundled payment modules were ERC20-only and rejected `msg.value`. The product needed either a native payment module or removal of native-token language from modules that could not collect it.

- Proof: `test_configure_TEMP_acceptsZeroTokenThenRejectsNativePayment` passed before the fix, showing ERC20 payment config accepted `address(0)` and then rejected native value.
- Regression: `test_configure_revertsForInvalidTokenAndRecipient`, `test_configure_revertsForInvalidRecipientAndInvalidBps`, `NativePaymentModuleTest`, and `test_mint_collectsNativePayment` cover early ERC20 zero-token rejection and native ETH collection.
- Fix: added `NativePaymentModule` for `Price.token == address(0)` flows and made ERC20 payment modules reject zero-token configuration.

### L-05: Oracle bounds and staleness defaults need production hardening

Status: Fixed.

`USDOracleRule` allows `maxStaleness = 0`, which disables freshness checks, and supports large decimal combinations. Production deployments should bound oracle decimals, token decimals, and mandatory staleness windows unless there is a concrete reason not to.

Temporary proof tests confirmed that a zero staleness window accepted year-old oracle answers and that large decimal combinations could reach unsafe arithmetic. The fix rejects zero `maxStaleness`, rejects token decimals above 18, rejects oracle decimals above 18 at configuration time, and rechecks oracle decimals during quote evaluation.

### L-06: Rounding dust can zero small prices or bias final split recipient

Status: Fixed / documented.

Percentage discounts and revenue splits use integer division. Very small prices can round to zero after discounts or allocate all dust to the final split recipient. This may be acceptable, but it should be deliberate and tested.

Temporary proof tests confirmed that a 1 bps discount on a 1-unit price made the mint free, a 1 bps markup on a 1-unit price stayed at 1, and split rounding remainder went to the final recipient. The controller now rounds `DISCOUNT_BPS` and `MARKUP_BPS` outputs up for non-zero results, while exact 100% discounts still produce zero. Split-payment dust remains explicit documented behavior and is covered by a regression test.
