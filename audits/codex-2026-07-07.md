## Verification Workflow

For each issue:

1. Write a temporary proof test that demonstrates the issue against the current code.
2. Run the proof test and record whether the issue is real.
3. If real, implement the smallest production fix that preserves the architecture.
4. Keep or convert the proof into a regression test.
5. Run focused tests, then broader verification.
6. Commit each meaningful change using a conventional commit message.

## Detailed Findings

### C-01: Malformed UTF-8 can pass emoji-only rules

Status: Fixed.

`LabelClassRule._nextCodepoint` decodes multibyte codepoints by masking each continuation byte with `0x3F`, but it does not verify that continuation bytes are actually in the UTF-8 continuation range `0x80..0xBF`. A malformed label can therefore be interpreted as a valid emoji codepoint by the rule engine even though it is not valid UTF-8.

Expected validation:

- Proof: `test_evaluateMint_TEMP_malformedUtf8BypassesEmojiRequirement` passed before the fix, showing malformed bytes `0xF0 0x5F 0x58 0x40` were priced as an emoji label.
- Regression: `test_evaluateMint_rejectsMalformedUtf8ThatWouldDecodeAsEmoji` now expects `InvalidUtf8Label`.
- Fix: `_nextCodepoint` now validates continuation bytes, rejects overlong encodings, rejects surrogate ranges, and rejects values above `U+10FFFF`.

### C-02: Renewal can use caller-selected activation for labels not governed by that activation

Status: Fixed.

`NamespaceController.renew` loads state from the activation registry and only rejects `AVAILABLE`. This allows two related paths:

- `RESERVED` labels can be renewed because ENSv2 `PermissionedRegistry.renew` accepts registered or reserved labels.
- If multiple activations share a registry, a caller can use a cheaper or less restrictive activation to renew a label originally minted under a stricter activation.

Expected validation:

- Proof: `test_renew_TEMP_extendsReservedLabel` passed before the fix, showing a `RESERVED` label expiry could be extended through the controller.
- Proof: `test_renew_TEMP_usesCheaperActivationForLabelMintedElsewhere` passed before the fix, showing a free activation could renew a paid-activation label without additional payment.
- Regression: `test_renew_revertsForReservedLabel` now expects `LabelNotRenewable` for `RESERVED` labels.
- Regression: `test_renew_revertsWhenActivationDidNotMintLabel` now expects `LabelActivationMismatch`.
- Fix: mints now record `labelActivations[registry][labelHash] = activationId`; renewals require `REGISTERED` status and a matching activation binding.

### C-03: Revoking registry admin does not stop active mint/renew

Status: Pending.

Activation lifecycle functions re-check that `activation.owner` still has root `ROLE_REGISTRAR_ADMIN`. Public `mint` and `renew` only check `activation.active`. If the owner loses registry admin rights while the controller still has registry mint/renew roles, the activation can remain live and unusable by the new admin except through global controller role revocation.

Expected validation:

- A proof test should create an activation, revoke the activation owner's registry admin role, then show `mint` or `renew` still succeeds.
- A fix should add a live authority check to public execution or provide a safe takeover/quarantine path for current registry admins.

### C-04: Flat pricing allows near-permanent names for one payment

Status: Pending.

`duration` is public caller input. `FixedPriceRule` and `USDOracleRule` can return a flat amount independent of duration, and the controller does not enforce a maximum duration. A buyer can choose a very large duration and receive a near-permanent expiry for the same configured price.

Expected validation:

- A proof test should mint or renew with a very large duration under flat pricing.
- The test should show price does not scale with duration.
- A fix should add activation-level duration bounds, require duration-scaled pricing for duration-sensitive sales, or make flat long-duration behavior an explicit activation option.

### C-05: Revoked activation owner can still pause or unpause

Status: Pending.

`PauseRule.setPaused` checks only `msg.sender == activation.owner`. It does not verify that the owner still has registry admin authority. This can allow a revoked owner to pause or unpause traffic after the registry owner intended to remove their authority.

Expected validation:

- A proof test should revoke registry admin authority and then show `setPaused` still succeeds.
- A fix should route pause authority through the controller or require a live authority check exposed by the controller.

### C-06: Fee-on-transfer ERC20s underpay while mint/renew succeeds

Status: Pending.

`ERC20PaymentModule` and `ERC20SplitPaymentModule` trust nominal `safeTransferFrom` success. Fee-on-transfer tokens can make the recipient receive less than `price.amount`, while the mint or renew operation still succeeds.

Expected validation:

- A proof test should use a fee-on-transfer mock token.
- The test should show mint or payment collection succeeds while recipients receive less than the quoted amount.
- A fix should either reject fee-on-transfer tokens by checking recipient balance deltas or document and constrain accepted payment tokens.

### C-07: ERC20 balance gates are flash-loanable

Status: Pending.

`TokenBalanceRule._evaluate` checks a live ERC20 balance. If the configured token is borrowable or otherwise temporarily transferable, a buyer can satisfy eligibility or receive discounts without durable ownership.

Expected validation:

- A proof test should show a temporary balance is sufficient to pass the rule.
- A fix may require a snapshot, staking/lockup, allowlisted credential source, or explicit documentation that this rule is only a live-balance convenience gate.

### C-08: Registrar-admin-only owners can monetize renewals

Status: Pending.

Activation creation checks only `ROLE_REGISTRAR_ADMIN`, while the controller itself needs `ROLE_REGISTRAR | ROLE_RENEW` on the registry. An account with registrar-admin authority but not renew-admin authority may configure renewal pricing and revenue collection through controller-held `ROLE_RENEW`.

Expected validation:

- A proof test should create an activation with an owner that has registrar admin but not renew admin, then renew through the controller.
- A fix should align activation owner authority with every registry power that the controller will exercise on behalf of that owner.

## Detailed Leads

### L-01: Unbound resolver node writes can target unrelated resolver nodes

Status: Pending lead.

The hooks compute resolver nodes from activation-provided `parentNode` and label hash. If an activation can set an unrelated `parentNode` and the hook has broad resolver write authority, minting in one registry could write records for another namespace's resolver node.

### L-02: Module approval revocation does not quarantine existing activations

Status: Pending lead.

Module approval is checked when storing activation module lists. Existing activations continue executing stored modules, and `updateModuleConfig` does not re-check approval. This may be intentional compatibility, but it is not an emergency-disable mechanism.

### L-03: Overlapping reservation claims let callers choose the least restrictive proof

Status: Pending lead.

`ReservationRule` verifies one caller-supplied claim. If the same root contains multiple valid claims for one label, callers can choose the claim that is most favorable. The root generation process must enforce uniqueness or the contract must encode precedence.

### L-04: Native pricing is documented but bundled payment modules reject native ETH

Status: Pending lead.

Several rule docs describe `token = address(0)` as native ETH pricing, but the bundled payment modules are ERC20-only and reject `msg.value`. The product should either include a native payment module or remove native-token language from modules that cannot collect it.

### L-05: Oracle bounds and staleness defaults need production hardening

Status: Pending lead.

`USDOracleRule` allows `maxStaleness = 0`, which disables freshness checks, and supports large decimal combinations. Production deployments should bound oracle decimals, token decimals, and mandatory staleness windows unless there is a concrete reason not to.

### L-06: Rounding dust can zero small prices or bias final split recipient

Status: Pending lead.

Percentage discounts and revenue splits use integer division. Very small prices can round to zero after discounts or allocate all dust to the final split recipient. This may be acceptable, but it should be deliberate and tested.
